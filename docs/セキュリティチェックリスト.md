### 3. `NEXT_PUBLIC_*` は公開情報として扱う

Next.js では `NEXT_PUBLIC_` で始まる環境変数はブラウザ側に埋め込まれる。

公開されてもよいもの:

- Clerk publishable key
- Stripe publishable key
- 公開 URL

公開してはいけないもの:

- secret key
- webhook secret
- DB URL
- Gmail app password
- 管理者メール一覧
- 内部テスト用 ID

確認ポイント:

- `NEXT_PUBLIC_TEST_USER_CLERK_ID` が本番に不要なら削除する。
- `NEXT_PUBLIC_APP_URL` は秘密ではないが、本番では `https://ele-cal.com` になっているか確認する。
- `STRIPE_SECRET_KEY` を誤って `NEXT_PUBLIC_STRIPE_SECRET_KEY` のような名前で入れていないか確認する。

### 4. Clerk middleware / proxy は全体保護ではない

`src/proxy.ts` では `clerkMiddleware()` を使っているが、コメント上も「全ページの認証強制ではなく、認証情報を付与するだけ」という設計になっている。

該当:

- `src/proxy.ts`

確認ポイント:

- `/admin` は `src/app/admin/layout.tsx` で守られている。
- `/api/admin/feedback/[id]` は `isCurrentUserAdmin()` で守られている。
- `/account`、`/subscribe`、Stripe portal 系 API は `auth()` で守られている。
- `/api/feedback` は匿名投稿を許可している。
- `/api/stripe-webhook` と `/api/clerk-webhook` は署名検証で守る設計。

注意:

- 依存関係の `@clerk/nextjs` に認可バイパス系 advisory が出ているため、今のコードが middleware で厳密保護していなくても、Clerk 更新は優先度が高い。
- 管理画面・管理 API は middleware だけに依存せず、現在のようにサーバー側で個別チェックする方がよい。

管理者判定は `ADMIN_EMAILS` に含まれるメールアドレスと Clerk のログインユーザーのメールを比較している。

良い点:

確認したい点:

- `ADMIN_EMAILS` は Vercel Sensitive ではなくてもよいが、管理者情報なので露出しない場所に置く。

### 6. 課金・有料機能制御はクライアント側が中心

有料機能の制限は `PaywallProvider` と各ページの `usePaywall()` / `requirePaid()` で行われている。

該当例:

- `src/components/PaywallProvider.tsx`
- `src/app/voltage_drop_v2/page.tsx`
- `src/app/breaker/page.tsx`
- `src/app/allowable_current/page.tsx`
- `src/app/pipe_size/page.tsx`
- `src/app/cable_rack/page.tsx`
- `src/app/apartment_main/page.tsx`

良い点:

- 課金状態は `src/app/layout.tsx` でサーバー側から `getSubscription(userId)` している。
- UI 上の有料操作は `requirePaid()` で止めている。

注意点:

- クライアント側の制御は、ユーザーの操作制限としては有効だが、秘密情報や本当に価値のある処理を守る境界にはならない。
- ブラウザに送られたコード・データは解析される可能性がある。
- もし有料ユーザーだけに見せたい計算ロジック・根拠データ・結果があるなら、サーバー API 側でも `auth()` と subscription check を行う必要がある。

確認ポイント:

- 有料機能の価値が「UI 操作の便利さ」なのか、「データや計算結果そのもの」なのかを整理する。
- 後者なら、サーバー側 API 化と課金チェックを検討する。
- 現在の freemium 設計で問題ない範囲なら、README や仕様書に「クライアント制御である」と明記する。
- `layout.tsx` の `getSubscription(userId)` がリクエストごとに再取得されているか確認する。
  Next.js App Router では DB 呼び出しがキャッシュされる場合があり、解約・支払い失敗・Webhook反映直後に
  課金状態が古い値のまま残る可能性がある。`noStore()` または `cache: 'no-store'` の設定を確認する。

### 8. Stripe Webhook は署名検証している

`/api/stripe-webhook` は `request.text()` で raw body を取り、`stripe.webhooks.constructEvent()` で `stripe-signature` と `STRIPE_WEBHOOK_SECRET` を検証している。

該当:

- `src/app/api/stripe-webhook/route.ts`

良い点:

- JSON parse 前に raw text を使っている。
- signature が無い場合は 400。
- `constructEvent()` で検証している。
- subscription created / updated / deleted、payment succeeded / failed を処理している。

確認ポイント:

- Stripe dashboard の Webhook endpoint URL が本番 URL になっているか。
- `STRIPE_WEBHOOK_SECRET` が、その endpoint 専用の `whsec_...` と一致しているか。
- Test mode と Live mode の webhook secret を混ぜていないか。
- Stripe CLI の webhook secret を本番に入れていないか。
- Webhook endpoint に送るイベントを必要最小限にしているか。
- Stripe dashboard の webhook delivery で失敗が残っていないか。

改善候補:

- Webhook は重複・遅延・順不同で届く前提にする。
- `event.id` を保存して重複処理を避けることを検討する。
- `customer.subscription.created` のメール送信は、再送時に重複通知になる可能性を確認する。
- 重要な状態更新では、Webhook payload だけでなく Stripe API から最新 subscription を取得して反映する設計も検討する。

### 10. フィードバック API は匿名投稿可能で、レート制限がない

`/api/feedback` はログインしていなくても投稿できる。

該当:

- `src/app/api/feedback/route.ts`
- `src/lib/db.ts`
- `src/lib/mail.ts`

良い点:

- category は allowlist。
- 本文は trim している。
- 本文は 4000 文字まで。
- SQL は Neon の tagged template を使っており、SQL injection には強い。
- メール送信失敗で投稿自体を失敗扱いにしない。

気になる点:

- 匿名投稿ができる。
- レート制限がない。
- captcha がない。
- honeypot がない。
- 投稿ごとに管理者メール通知が飛ぶため、メール通知の乱用や Gmail 制限に当たる可能性がある。
- 本文をそのまま DB とメールに保存・送信するため、個人情報や機密情報が投稿される可能性がある。

確認ポイント:

- 匿名投稿を許可する業務上の理由があるか。
- ログイン必須にするか。
- 匿名のままなら、Vercel Firewall / rate limiting / Turnstile / captcha / honeypot を検討する。
- メール通知は短時間に大量発生しないよう制御する。
- DB に保存する個人情報の範囲をプライバシーポリシーと照合する。

### 11. DB 設計で確認したいこと

`feedback_requests` と `feedback_replies` の migration は docs にある。

該当:

- `docs/migration-feedback.sql`
- `docs/migration-feedback-status.sql`
- `src/lib/db.ts`

良い点:

- `feedback_requests.category` は CHECK 制約あり。
- `feedback_requests.status` は CHECK 制約あり。
- `feedback_replies.request_id` は foreign key あり。
- SQL は tagged template で実行している。

確認したい点:

- `subscriptions` テーブルの migration が docs に見当たらないため、実 DB の制約を確認する。
- `subscriptions.clerk_user_id` に UNIQUE があるか。
- `subscriptions.stripe_customer_id` に UNIQUE または index があるか。
- `ON CONFLICT (clerk_user_id)` が実 DB で成立しているか。
- `UPDATE subscriptions WHERE stripe_customer_id = ...` が複数行に当たらないか。
- RLS を使っていない場合、アプリ側認可が唯一の境界になることを理解する。

確認 SQL 例:

```sql
SELECT table_name
FROM information_schema.tables
WHERE table_schema = 'public'
ORDER BY table_name;

SELECT indexname, indexdef
FROM pg_indexes
WHERE schemaname = 'public'
  AND tablename IN ('subscriptions', 'feedback_requests', 'feedback_replies')
ORDER BY tablename, indexname;

SELECT conname, pg_get_constraintdef(oid)
FROM pg_constraint
WHERE conrelid IN (
  'subscriptions'::regclass,
  'feedback_requests'::regclass,
  'feedback_replies'::regclass
);
```

これは直接の脆弱性ではないが、以下のリスクがある。

### 15. GitHubリポジトリのハードニング（A03 対応）

リポジトリが Public なため、コードは全世界に公開されている。

確認ポイント:

- `.gitignore` に `.env.local` が含まれているか。
- シークレットが過去に一度でもコミットされていないか。コミットメッセージと過去のファイル内容の両方を確認する。

  コミットメッセージ検索:
  ```powershell
  git log --grep="SECRET|PASSWORD|KEY|TOKEN" --all
  ```

  過去のファイル内容検索（.env 系ファイルが履歴に混入していないか）:
  ```powershell
  git log --all --full-history -- ".env*"
  git log --all --full-history -- "*.env"
  ```

  特定のコミット内容を確認する場合:
  ```powershell
  git show <commit-hash> -- .env.local
  ```

- main ブランチへの直接プッシュ保護（Branch protection rules）が設定されているか。
  GitHub > Settings > Branches > Add rule で確認。
- GitHub の Secret scanning が有効か（Settings > Code security and analysis）。
- Dependabot アラートが有効か（同箇所）。
- `package-lock.json` がコミットされているか（依存関係の再現性確保）。
