# 設計判断ログ

> 「なぜそう実装したか」を1判断1セクションで記録する。
> 後で覆す場合も**古いエントリは消さない**。「却下」マークを付けて新エントリを追加。
>
> フォーマット：
> ```
> ## D-NNN: タイトル
> - **日付**: YYYY-MM-DD
> - **状態**: 採用 / 却下 / 保留
> - **背景**: どんな問題があったか
> - **判断**: 何を選んだか
> - **理由**: なぜそれを選んだか
> - **代替案**: 検討して却下した案
> - **影響**: この判断によって決まること
> ```

---

## D-001: 入力単位を kW に統一（過去判断・要再検討）

- **日付**: （初期実装時。正確な日付は git history 参照）
- **状態**: 採用中（要再検討）
- **背景**: 一般負荷（LED照明など）の消費電力は数W単位で、kW表記だと「0.005kW」と入力する必要があり不便
- **判断**: 入力単位を kW で統一
- **理由**: **内線規程が kW 表記**で記載されているため、参照側との整合性を優先
- **代替案**: W 表記 / 負荷種別ごとに W・kW を切替
- **影響**: 一般負荷の入力体験が悪い。お客さんからフィードバックあり（やること一覧.md フェーズ2）

## D-002: 負荷種別の混在を許容（過去判断・変更予定）

- **日付**: （初期実装時）
- **状態**: 採用中（変更予定）
- **背景**: 1台のブレーカーに複数の負荷をぶら下げる構成を表現したい
- **判断**: 負荷リストに「一般負荷 / モーター / 溶接機」を**混在して追加可能**にした
- **理由**: 実装の単純化。「複数負荷の合算」というモデルとして自然
- **代替案**: 種別ごとに排他（最初に選んだ種別しか追加できない）
- **影響**: 現場の慣習と乖離（現場では一般負荷と溶接機を同じブレーカーに乗せない）。
  → やること一覧.md フェーズ1③で**排他選択に変更予定**
